# LocalGitLabGateway 通信方式設計

## 1. 目的

本書は、LocalGitLabGateway における Public Gateway Service と Private Bridge Agent の通信方式を定義する。

## 2. 前提条件

- Public Gateway Host と Private GitLab Host のサーバー費用以外は無償であること
- 長期運用の安定性を優先すること
- GitLab 本体をインターネットへ直接公開しないこと
- ChatGPT からの操作入口は Public Gateway Service に限定すること
- GitLab 認証情報は Private GitLab Host 側にのみ保持すること
- Public Gateway Host 側に GitLab Personal Access Token を保存しないこと

## 3. 採用方式

LocalGitLabGateway の初期採用方式は、以下とする。

```text
ChatGPT
  ↓ HTTPS
Public Gateway Service
  ↓ WireGuard VPN
Private Bridge Agent
  ↓ GitLab REST API v4
Managed GitLab
```

## 4. 採用理由

| 観点 | 評価 |
| --- | --- |
| 無償性 | WireGuard、FastAPI、nginx、systemd、SQLite は無償で利用できる構成にできる |
| 安定性 | WireGuard は常時接続 VPN として安定運用しやすい |
| セキュリティ | GitLab を外部公開せず、Public Gateway Service から Private Bridge Agent への到達経路を VPN 内に限定できる |
| 実装容易性 | Public Gateway Service と Private Bridge Agent を通常の HTTP API として設計できる |
| 運用容易性 | Linux の systemd、journalctl、nginx と組み合わせて管理できる |

## 5. 構成要素

## 5.1 Public Gateway Host

Public Gateway Host は、インターネットから HTTPS で到達可能な Linux サーバーである。

Public Gateway Host では、以下を稼働させる。

- nginx
- Public Gateway Service
- WireGuard peer
- systemd unit
- 監査ログ出力機構

## 5.2 Public Gateway Service

Public Gateway Service は、ChatGPT からの要求を受け付ける公開側 API サービスである。

Public Gateway Service は、以下を担当する。

- ChatGPT からの HTTPS 要求受付
- API Key 認証
- 要求スキーマ検証
- 操作 allowlist 検証
- request_id 採番
- 監査ログ記録
- Private Bridge Agent への転送
- Private Bridge Agent からの応答整形

## 5.3 Private GitLab Host

Private GitLab Host は、ファイアウォール内部に配置された Linux サーバーである。

Private GitLab Host では、以下を稼働させる。

- Managed GitLab
- Private Bridge Agent
- WireGuard peer
- systemd unit

## 5.4 Private Bridge Agent

Private Bridge Agent は、Managed GitLab への操作を代理実行する内部側エージェントである。

Private Bridge Agent は、以下を担当する。

- Public Gateway Service からの VPN 内要求受付
- 内部 API Key または mTLS による認証
- GitLab REST API v4 呼び出し
- GitLab 認証情報の内部保持
- GitLab 応答の正規化
- 実行結果返却

## 6. 通信経路

## 6.1 ChatGPT から Public Gateway Service

| 項目 | 方針 |
| --- | --- |
| 通信方式 | HTTPS |
| 認証 | API Key |
| 公開範囲 | インターネット公開 |
| 終端 | nginx |
| アプリケーション | FastAPI |

## 6.2 Public Gateway Service から Private Bridge Agent

| 項目 | 方針 |
| --- | --- |
| 通信方式 | HTTP または HTTPS |
| 通信経路 | WireGuard VPN 内 |
| 認証 | 内部 API Key を初期案とする |
| 公開範囲 | VPN 内限定 |
| アプリケーション | FastAPI |

## 6.3 Private Bridge Agent から Managed GitLab

| 項目 | 方針 |
| --- | --- |
| 通信方式 | HTTP または HTTPS |
| 接続先 | GitLab REST API v4 |
| 認証 | GitLab Personal Access Token |
| トークン保持場所 | Private GitLab Host のみ |

## 7. 初期ポート設計

| 用途 | ホスト | ポート例 | 公開範囲 |
| --- | --- | --- | --- |
| HTTPS | Public Gateway Host | 443/tcp | インターネット |
| WireGuard | Public Gateway Host | 51820/udp | Private GitLab Host から到達可能な範囲 |
| Public Gateway Service | Public Gateway Host | 127.0.0.1:8080 | ローカルのみ |
| Private Bridge Agent | Private GitLab Host | WireGuard IP:8081 | VPN内のみ |
| Managed GitLab | Private GitLab Host | 127.0.0.1 または LAN IP | 内部のみ |

## 8. セキュリティ境界

## 8.1 Public Gateway Host に置いてよい情報

- ChatGPT から Public Gateway Service へ接続するための API Key ハッシュ
- 操作 allowlist
- 監査ログ
- Private Bridge Agent の VPN 内 URL

## 8.2 Public Gateway Host に置いてはならない情報

- GitLab Personal Access Token
- GitLab 管理者認証情報
- GitLab ユーザーパスワード
- CI/CD secret
- Private GitLab Host の不要な内部情報

## 8.3 Private GitLab Host に置く情報

- GitLab Personal Access Token
- Managed GitLab URL
- Private Bridge Agent 内部認証情報
- Public Gateway Service からの許可情報

## 9. 初期実装方針

## 9.1 Public Gateway Service

初期実装では、以下の API を提供する。

- `GET /health`
- `GET /v1/projects`
- `GET /v1/projects/{project_id}/branches`
- `GET /v1/projects/{project_id}/files/{file_path}`
- `POST /v1/projects/{project_id}/files/{file_path}`
- `PUT /v1/projects/{project_id}/files/{file_path}`
- `POST /v1/projects/{project_id}/merge-requests`
- `POST /v1/projects/{project_id}/issues`

## 9.2 Private Bridge Agent

初期実装では、Public Gateway Service と同等の内部 API を提供し、GitLab REST API v4 へ変換する。

## 10. フォールバック方針

WireGuard 構成が利用できない環境では、第二候補として Reverse WebSocket 方式を検討する。

ただし、初期設計および初期実装では WireGuard 方式を正とする。

## 11. 除外方式

## 11.1 Tailscale

Tailscale は構築容易性が高いが、外部サービス依存があるため初期採用しない。

## 11.2 商用トンネルサービス

Cloudflare Tunnel などの商用または外部SaaS依存の方式は、サーバー費用以外を無償とする条件に対して将来制約が発生する可能性があるため初期採用しない。

## 11.3 SSH Reverse Tunnel

SSH Reverse Tunnel は安定性が高いが、APIサービス間通信としての責務分離が曖昧になりやすいため初期採用しない。

## 12. 決定事項

- 初期通信方式は WireGuard + HTTPS API とする
- ChatGPT からの入口は Public Gateway Service の HTTPS API に限定する
- Private Bridge Agent は VPN 内からのみ到達可能とする
- GitLab Personal Access Token は Private GitLab Host にのみ保存する
- Public Gateway Host は GitLab 認証情報を保持しない
